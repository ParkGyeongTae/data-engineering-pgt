# 현업에서는 언제 Zeppelin을 쓸까?

## 들어가며

Apache Zeppelin에 기여를 하려고 하다 보면 코드는 눈에 익지만, 실제 현업에서 어떻게 사용하는지는 잘 그려지지 않을 수 있다. Data Engineer로 커리어를 시작해 현재 Data Analytics Engineer로 일하면서 겪은 개인적인 경험을 바탕으로 이 글을 정리했다. 회사마다 규모·조직·도메인이 달라 정답은 아니지만, 최대한 일반적인 내용으로 구성했다.

데이터플랫폼이 어떻게 돌아가는지 보면, Zeppelin이 언제 쓰이는지 자연스럽게 이해할 수 있다. 이 글에서는 다음 세 가지를 다룬다.

1. [수집] 운영계 데이터를 분석계로 복제
2. [처리] 로컬 파이썬의 한계
3. [조회] Athena, Superset, Tableau, Hue… 그중 Zeppelin을 선택하는 순간

## 1. 운영계 데이터를 분석계로 복제

### 문제

- 백엔드에서는 일반적으로 데이터를 RDB에 저장한다. 행 단위 처리가 중요하기 때문에 합리적인 선택이다.
- 분석하려는 데이터도 결국 이 운영 DB에 있다.
- 서비스에 연결된 운영 DB에 분석 쿼리를 직접 날리면 부하·락·장애 전파로 이어질 수 있다.
- 분석 쿼리는 한 명이 아니라 전체 사용자·특정 집단·특정 페이지 단위로 조회하기 때문에, 조회만으로도 부하가 생긴다.

### 해결

- 운영계 데이터를 S3 같은 별도 저장소로 복제/적재해서 "분석용 데이터가 준비되는 곳"을 따로 둔다.
- 어느 정도 규모의 회사는 이 복제를 데이터 엔지니어가 담당한다. 두 데이터베이스 간 동기화, 나아가 정합을 맞추는 업무까지 포함한다.
- 그냥 파일로만 쌓아두면 못 쓴다. Hive 형식(스키마 + 파티션)으로 저장해서, RDB를 쓰듯 쿼리로 데이터를 조회할 수 있게 유지한다.
- 별도 경로도 있다. 로그성 데이터는 클라우드 스토리지 비용이 경제적이라 RDB를 거치지 않고 S3에 바로 적재하는 것도 합리적인 판단이다.

다음 질문은, 그렇게 쌓인 데이터를 어떻게 처리할 것인가다.

## 2. 로컬 파이썬의 한계

- S3에 데이터가 쌓이기 시작하면 처음엔 로컬 pandas로 충분히 처리 가능하다.
- pandas는 데이터를 메모리에 전부 올려서 처리하기 때문에, 데이터가 커지면 메모리 부족(OOM)으로 아예 처리가 안 되는 시점이 온다.
- 메모리를 넘기지 않아도, 단일 프로세스라 처리 시간이 몇 시간씩 걸리는 배치가 생기기 시작한다.
- Spark는 데이터를 여러 노드에 나눠서 분산 처리함으로써 메모리·처리 시간 문제를 동시에 해결한다.
- EMR은 그 Spark 클러스터를 직접 구축·운영하지 않고 관리형으로 띄우고 내릴 수 있게 해주는 인프라다.

다음 질문은, 처리된 큰 데이터를 어떻게 조회·조작할 것인가다.

## 3. 조회 도구 비교

이 데이터를 조회하는 사람도 다양하다.

- 분석계에 존재하는 데이터를 조회하는 개발자
- 푸시 성과를 빠르게 확인하고 싶은 PM
- 요청받은 분석 업무를 해결하려는 데이터 분석가 등등

같은 데이터도 조회하는 사람·목적에 따라 쓰는 도구가 다르다.

- **Athena**: SQL로 가볍게 조회
- **Superset / Tableau**: 대시보드·BI 리포트
- **Hue**: 하둡 생태계 쿼리 실행
- **Jupyter**: 범용 노트북. 로컬/개인 분석에 강하지만 Spark 연동과 협업 공유는 상대적으로 약함
- **Databricks**: 관리형 Spark 노트북 플랫폼. Zeppelin+EMR 조합과 가장 직접적으로 경쟁하는 상용 대안
- **Zeppelin**: Spark 노트북 기반, 코드+시각화 결합, 탐색적 분석 과정을 팀과 공유

### Zeppelin을 선택하는 순간

- 로컬 파이썬(Jupyter 포함)은 성능뿐 아니라 접근 권한 문제도 있다. 개인이 AWS 접근 권한을 가지고 직접 다뤄야 해서, 데이터 엔지니어가 아닌 사람에게까지 권한이 퍼지며 보안·거버넌스 부담이 커진다.
- Zeppelin 같은 공유 노트북 플랫폼은 개인 AWS 키 없이 접근을 중앙에서 관리해서 이 문제를 해결한다.
- 가장 가까운 대안인 Databricks와 비교하면, Zeppelin+EMR 조합은 오픈소스라 별도 라이선스 비용 없이 필요할 때만 클러스터를 띄우고 내릴 수 있다. 비용·거버넌스를 조직이 직접 통제하고 싶을 때 선택지가 된다.
- 정리하면 Zeppelin은 "Spark 연동이 필요한 규모 + 접근 권한의 중앙 관리 + 비용 통제"가 동시에 요구되는 지점에서 선택된다.

## 마무리

오늘 흐름을 다시 보면: 운영 DB 복제 → Spark/EMR로 분산 처리 → 처리된 데이터를 조회하는 여러 방식 중 하나로 Zeppelin이 선택된다.
